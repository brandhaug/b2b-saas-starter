# Shared module-aware env validation

The starter includes a shared environment validation package that understands baseline Cloudflare configuration and optional provider module configuration. It should expose redacted configuration status so starter module readiness can explain missing setup without leaking secrets in UI, logs, reports, or errors.

**Status (2026-07-02):** wired end-to-end. `readServerEnv` + `moduleConfigStatus` (`packages/env/src/server.ts`) run over the real worker env in `apps/web/src/lib/capabilities.ts` and `apps/api/src/index.ts`; the resulting statuses are passed as `StarterEnv.moduleConfig` and overlaid onto `StarterModuleCatalog` and `IntegrationSurfaces` by `withModuleEnvStatus` (`packages/capabilities/src/layers.ts`). A module with unset env vars reports `needs-config` (with the missing var names, never values) regardless of stored fixture state; env present but runtime unwired (billing, observability) reports `attention`. With no env at all the app still boots provider-light — `readServerEnv` defaults the required baseline vars outside strict mode.

**Status (2026-07-02, catalog consolidation):** the var catalog now lives in ONE place. `ServerEnvSchema` is the single source: `readServerEnv` accepts a raw worker env (`Record<string, unknown>`, no casts or remaps) and picks/decodes the schema's own keys; `makeStarterEnvModuleConfig(env)` packages the full recipe the workers copy-pasted; the exported `optionalModuleEnvSecretKeys`/`optionalModuleEnvPlainKeys` lists drive alchemy's env forwarding; and `apps/web/src/worker-env.d.ts` derives its string vars from `ServerEnv`. The `EMAIL_FROM_ADDRESS` alias was eliminated — every layer ships `CLOUDFLARE_EMAIL_FROM`. `PUBLIC_SITE_URL` (never read, never provisioned) was removed from the schema. AI vars (`WORKERS_AI_ENABLED`, `OPENAI_API_KEY`, `OPENAI_BASE_URL`, `OPENAI_MODEL_ID`) are schema fields with an `ai` module status entry.
