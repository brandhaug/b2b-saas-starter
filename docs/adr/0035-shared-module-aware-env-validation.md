# Shared module-aware env validation

The starter includes a shared environment validation package (`packages/env`) that understands baseline Cloudflare configuration and optional provider module configuration, and exposes redacted configuration status so starter module readiness can explain missing setup without leaking secrets in UI, logs, reports, or errors.

`ServerEnvSchema` in `packages/env/src/server.ts` is the single source for the variable catalog. `readServerEnv` accepts a raw worker env and picks/decodes the schema's own keys, `moduleConfigStatus` derives per-module status from it, and `makeStarterEnvModuleConfig(env)` packages the recipe both workers use (`apps/web/src/lib/capabilities.ts`, `apps/api/src/index.ts`). The exported `optionalModuleEnvSecretKeys` and `optionalModuleEnvPlainKeys` lists drive Alchemy's env forwarding, and `apps/web/src/worker-env.d.ts` derives its string vars from `ServerEnv`. Email config ships as `CLOUDFLARE_EMAIL_FROM` on every layer (no alias), and the AI vars (`WORKERS_AI_ENABLED`, `OPENAI_API_KEY`, `OPENAI_BASE_URL`, `OPENAI_MODEL_ID`) are schema fields behind an `ai` module status entry.

Statuses are passed as `StarterEnv.moduleConfig` and overlaid onto `StarterModuleCatalog` and `IntegrationSurfaces` by `withModuleEnvStatus` (`packages/capabilities/src/layers.ts`). A module with unset env vars reports `needs-config` with the missing var names (never values) regardless of stored fixture state, while env that is present but runtime-unwired (billing, observability) reports `attention`. With no env at all the app still boots provider-light: `readServerEnv` defaults the required baseline vars outside strict mode.
